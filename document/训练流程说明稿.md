# 训练流程说明稿

## 一、系统中的模型配置概览

本系统并非只依赖单一模型，而是由多类模块共同组成完整检测链路。按照实际运行逻辑，可分为人脸检测模块、视觉特征提取模块、融合判别模块、语义分析模块与策略选择模块。

人脸检测模块配置了三级方案。第一层为 `InsightFaceDetector`，负责输出人脸框、关键点与 embedding，是优先级最高的人脸检测方案；第二层为 `OpenCVHaarDetector`，用于在深度学习检测器不可用时完成传统人脸检测；第三层为 `HeuristicFaceDetector`，在前两级均不可用时，以候选区域启发式评分方式完成兜底检测。对应实现位于 `backend/app/services/backends.py`。

视觉特征提取模块配置了两级方案。主方案为训练后的 `EfficientNet-B0` 视觉模型，由 `EfficientNetVisualBackend` 负责加载与调用；回退方案为 `HeuristicVisualBackend`，基于压缩痕迹、频域异常、一致性与模糊度等启发式信号给出视觉分数。二者统一由 `VisualBackendManager` 调度。

融合判别模块同样采用两级结构。主方案为训练后的 `XGBoost` 融合模型，由 `XGBoostFusionBackend` 负责加载并对结构化特征向量进行预测；回退方案为 `HeuristicFusionBackend`，对视觉分数、质量分数、压缩分数、语义分数等特征执行人工权重组合。二者由 `FusionBackendManager` 调度。

语义分析模块配置了三级方案。优先方案为远程视觉大模型接口 `QwenVisionSemanticBackend`，次优方案为本地视觉语言模型 `LocalQwenSemanticBackend`，兜底方案为 `RuleBasedSemanticBackend`。策略选择模块则由 `QwenStrategyBackend` 负责大模型推理，若不可用则退回规则策略逻辑。

从训练与接入角度看，本项目实际亲自完成训练并接入系统的核心模型有两个：一是 `EfficientNet-B0` 视觉轻量模型，二是 `XGBoost` 融合模型。其余模块主要承担检测、回退和语义增强职责。

## 二、视觉轻量模型的训练流程

### 1. 原始数据来源与选择理由

视觉轻量模型的主训练数据来源是 `FaceForensics++`。项目没有直接使用整个公开数据仓库的全部内容，而是通过下载脚本有选择地获取最适合图像级 Deepfake 检测训练的子集，包括：

1. `original`
2. `Deepfakes`
3. `Face2Face`
4. `FaceSwap`
5. `NeuralTextures`

下载脚本为 `backend/scripts/download_ffpp_subset.py`。该脚本默认压缩级别设置为 `c23`，并明确将其作为本项目推荐配置。采用 `c23` 的原因在于：其相较于原始无压缩视频更接近实际网络传播场景，又比极高压缩级别更适合作为初始训练数据，能够在数据真实性与训练稳定性之间取得平衡。

该脚本还保留了 `DeepFakeDetection` 等可选数据源入口，用于扩展训练或后续泛化测试，但当前核心训练流程优先围绕 FF++ 主子集展开。

### 2. 从视频到图像的抽帧策略

视觉模型并不是直接用视频序列训练，而是先从视频中抽取图像帧，再将图像整理为标准分类数据集。抽帧脚本为 `backend/scripts/extract_ffpp_frames.py`。

项目采用的是稀疏均匀抽帧，而不是逐帧全量抽取。默认参数为：

1. `--fps 1.0`
2. `--max-frames-per-video 40`

脚本的具体逻辑是先读取原视频帧率 `source_fps`，再按 `frame_interval = round(source_fps / target_fps)` 计算采样间隔，最后按固定步长保存帧。这样做的主要原因有三点：

1. 相邻视频帧通常高度相似，逐帧展开会造成严重冗余；
2. 稀疏均匀采样能够覆盖视频中不同时间位置的伪造痕迹；
3. 在毕业设计场景下，该策略能够有效控制存储规模和训练成本。

因此，当前项目采用的并不是复杂关键帧提取，而是更强调工程稳定性和可重复性的统一抽帧方案。

### 3. 从抽帧结果到可训练目录的整理过程

抽帧完成后，并不能直接送入 `torchvision.datasets.ImageFolder` 训练，还需要进一步整理成监督学习目录。负责这一工作的脚本是 `backend/scripts/prepare_visual_dataset.py`。

该脚本将：

1. `original` 目录下的所有图像归为 `real`
2. `Deepfakes`、`Face2Face`、`FaceSwap`、`NeuralTextures` 等目录下的图像归为 `fake`

随后脚本按设定的 `val_ratio` 对样本进行随机划分，并整理成如下结构：

1. `train/real`
2. `train/fake`
3. `val/real`
4. `val/fake`

为了提升大规模数据整理效率，脚本优先采用硬链接方式物化训练集，若系统不支持硬链接，则自动回退为复制。这样既节省空间，也能兼顾不同环境的兼容性。

这一阶段体现的并不是简单的数据搬运，而是完整的数据工程工作：从原始公开视频序列出发，将其转化为深度学习训练脚本可直接消费的标准分类数据集。

### 4. 训练脚本如何构建数据加载流程

训练脚本位于 `backend/scripts/train_visual_model.py`。其总体思路是利用 `ImageFolder` 和 `DataLoader` 构建标准二分类训练流程。

在 `build_dataloaders()` 中，脚本分别读取 `train` 和 `val` 两个目录，并对图像施加如下预处理：

1. `Resize((256, 256))`
2. `ToTensor()`
3. `Normalize(mean=[0.485, 0.456, 0.406], std=[0.229, 0.224, 0.225])`

这种处理方式与常见 ImageNet 预训练模型保持一致，能够让预训练骨干的特征分布迁移更稳定。

### 5. 轻量视觉模型的结构配置

视觉主模型采用的是 `EfficientNet-B0`。训练脚本首先加载 `models.efficientnet_b0(weights=models.EfficientNet_B0_Weights.DEFAULT)`，说明该模型使用了 ImageNet 预训练权重作为初始化，而不是随机初始化。

随后，脚本将默认分类头替换成二分类线性层：

1. 获取 `model.classifier[1].in_features`
2. 构造新的 `nn.Linear(..., 2)`
3. 将输出类别调整为 `real/fake`

该配置体现出两个关键点：

1. 模型本身是轻量级卷积网络，适合服务化部署；
2. 训练方式是迁移学习而非从零训练，能够在中小规模数据集上更快收敛。

### 6. 训练过程的具体执行逻辑

训练过程中，脚本按 epoch 进行标准监督学习迭代：

1. 从训练集读取批次图像与标签；
2. 将张量送入设备（GPU 或 CPU）；
3. 执行前向传播得到分类 logits；
4. 使用 `CrossEntropyLoss` 计算损失；
5. 使用 `AdamW` 更新参数；
6. 每轮训练结束后，在验证集上评估 `accuracy`、`f1`、`auc`；
7. 若当前轮验证精度优于历史最优，则保存权重。

这里的训练过程不是“固定训练完就覆盖保存”，而是采用了 best checkpoint 保存机制，只保留验证表现最优的一次模型权重，这能降低过拟合阶段模型被错误覆盖的风险。

### 7. 使用的优化方法与原因

视觉轻量模型训练中的关键优化方法包括以下几项：

#### 预训练初始化

项目没有从随机权重起训，而是使用 `EfficientNet_B0_Weights.DEFAULT` 进行迁移学习。这样做的价值在于：

1. 降低训练早期的不稳定性；
2. 更快学习到通用纹理、边缘和颜色统计特征；
3. 减少训练时间与算力成本。

#### AdamW 优化器

训练脚本使用的是 `torch.optim.AdamW`。相比最基础的 SGD，AdamW 在当前这种图像级分类任务中通常更容易稳定收敛，并且对权重衰减的处理更规范，适合毕业设计阶段的高效实验与部署。

#### CrossEntropyLoss 损失函数

视觉模型任务本质上是单标签二分类，因此使用 `nn.CrossEntropyLoss()` 作为标准监督学习损失函数是合理且直接的选择。

#### 自动设备选择

脚本中通过 `resolve_device()` 自动选择 `cuda` 或 `cpu`。这使得同一份训练脚本既可在服务器 GPU 环境运行，也可在本地 CPU 环境调试。

#### 验证集驱动的阈值冻结

项目的最终目标不是简单输出一个二分类概率，而是要服务于系统中的“三态输出”：疑似真实、疑似伪造、需人工复核。因此训练脚本在模型训练完成后，并没有直接固化 `0.5` 作为判定阈值，而是调用 `calibrate_thresholds()` 在验证集分数上搜索最优 `true_threshold` 与 `fake_threshold`。

该函数会遍历候选阈值对，并以分类准确率优先、复核率尽量低为原则，选出更适合系统业务输出形式的阈值区间。最终，这两个阈值会与训练指标一起写入 metadata 文件，用于后端运行时加载。

### 8. 训练产物的生成方式

视觉模型训练完成后会产出两类核心资产：

1. `efficientnet_b0_deepfake.pt`
2. `efficientnet_b0_deepfake.meta.json`

其中，权重文件负责模型推理；metadata 文件负责描述：

1. 模型名称
2. 生成时间
3. 数据版本
4. 输入尺寸
5. 类别分布
6. 验证指标
7. 阈值与阈值来源
8. 训练配置

这种“权重 + metadata”的模型资产组织方式，保证了训练结果不只是一个孤立的 `.pt` 文件，而是可以被后端系统完整识别、加载、展示和追踪的部署单元。

## 三、融合模型的训练流程

### 1. 为什么还需要训练融合模型

视觉模型输出的是单脸或单图像层面的深伪分数，但系统实际掌握的信息并不只有这一项。除了视觉判别分数外，系统还会同时产生：

1. 多张人脸中的最大分数与均值分数
2. DCT 频域异常分数
3. 图像质量分数
4. 增强一致性分数
5. 压缩痕迹分数
6. 人脸数量
7. 语义分数

如果只依赖视觉模型，很难充分利用这些异构特征。因此系统进一步训练了一个 `XGBoost` 融合模型，用于学习多源特征之间的组合关系，从而提升最终判决的稳定性和结构化程度。

### 2. 融合模型训练数据如何生成

融合模型训练的关键难点不在于写一个 `XGBoostClassifier`，而在于如何把系统的中间结果变成标准训练样本。对此，项目专门编写了离线特征导出脚本 `backend/scripts/export_fusion_features.py`。

该脚本不会凭空制造训练数据，而是复用与线上推理高度一致的处理链路：

1. 通过 `iter_examples()` 遍历离线图像样本；
2. 调用 `build_offline_feature_payload()` 对每张图执行人脸检测、视觉分析、图库匹配；
3. 从每张图得到若干 `FaceResult`；
4. 再通过 `aggregate_fusion_features()` 将逐脸特征聚合为图像级结构化特征行；
5. 最终写出 JSONL 文件，供 XGBoost 训练。

这说明融合模型并非脱离主系统单独存在，而是直接建立在系统特征提取流水线之上。

### 3. 融合模型的输入特征构成

训练脚本 `backend/scripts/train_fusion_model.py` 中定义的 `FEATURE_KEYS` 包含 8 个特征：

1. `max_face_score`
2. `mean_face_score`
3. `dct_mean`
4. `quality_mean`
5. `consistency_mean`
6. `compression_mean`
7. `face_count`
8. `semantic_score`

这些特征覆盖了视觉风险、频域统计、图像质量、结构一致性、压缩痕迹与语义信息等多个层面，体现出融合模型的作用并不是替代视觉模型，而是在其上方做结构化证据整合。

需要说明的是，当前离线特征导出过程中，`semantic_score` 使用的是 `0.0` 占位值，主要原因在于批量离线导出真实语义特征会显著增加时间成本和外部依赖不确定性。因此，当前版本中 XGBoost 训练主要学习的是视觉和统计特征的组合关系，而在线推理阶段则保留真实语义分析的增强能力。

### 4. XGBoost 训练脚本的执行过程

`train_fusion_model.py` 的训练流程大致分为以下几步：

1. 读取训练 JSONL 与可选验证 JSONL；
2. 逐行检查是否包含全部特征键以及标签字段；
3. 将每条样本转换为 `numpy` 特征矩阵和标签数组；
4. 初始化 `xgb.XGBClassifier`；
5. 在训练集上完成拟合；
6. 在验证集上计算 `accuracy`、`f1`、`auc`；
7. 若有验证集，则进一步执行阈值校准；
8. 保存模型文件和 metadata 文件。

### 5. 融合模型采用的超参数与其含义

当前脚本中配置的主要超参数包括：

1. `n_estimators = 120`
2. `max_depth = 4`
3. `learning_rate = 0.08`
4. `subsample = 0.9`
5. `colsample_bytree = 0.9`
6. `eval_metric = "logloss"`

这些配置体现出比较典型的中小规模结构化特征训练思路：

1. 通过适度数量的 boosting 轮数提升表达能力；
2. 通过受控树深度抑制过拟合；
3. 通过较小学习率提高训练稳定性；
4. 通过样本与特征的随机采样增强泛化能力。

### 6. 融合模型训练中的优化思路

#### 多特征而非单特征输入

融合模型并不是学习一个简单的视觉分数阈值，而是同时观察 8 个来源不同的结构化信号，这使得模型能够学习到更复杂的非线性判别边界。

#### 验证集驱动的阈值校准

与视觉模型一致，融合模型在训练后也会对 `true_threshold` 和 `fake_threshold` 进行校准。这是因为后端系统最终输出的是三态标签而不是简单二分类结果，因此必须将模型概率映射到“真实 / 灰区复核 / 伪造”的业务语义空间中。

#### 元数据固化

XGBoost 训练结果同样不是只保留一个模型文件，而是输出：

1. `xgboost_fusion.json`
2. `xgboost_fusion.meta.json`

metadata 中会记录：

1. 模型名称
2. 数据版本
3. 特征键顺序
4. 训练样本数与验证样本数
5. 评估指标
6. 阈值及阈值来源

这样能够保证后端在推理时按完全一致的特征顺序进行调用，避免“训练与部署特征错位”的问题。

## 四、训练后的模型是如何接入系统的

视觉模型在运行时由 `EfficientNetVisualBackend` 加载。系统会先检查权重文件路径和 metadata 文件路径是否存在，若二者完整，则创建与训练时一致的 `EfficientNet-B0` 网络结构，加载训练好的 `state_dict`，并在单张人脸裁剪图上进行前向推理。

融合模型在运行时由 `XGBoostFusionBackend` 加载。系统会检查 `xgboost_fusion.json` 与对应 metadata 是否存在，若完整，则在最终决策阶段使用 8 维结构化特征向量完成图像级融合预测。

模型接入不是简单“文件放进去就结束”，而是通过 `model_registry.py` 统一管理模型是否就绪、metadata 是否存在、阈值来自何处、前端应展示哪一个模型版本信息。运行时阈值优先读取融合模型 metadata，其次读取视觉模型 metadata，最后才退回默认配置。

## 五、训练流程的工程工作量总结

从工程角度看，这一套训练流程的工作量并不只是“运行了一次训练命令”，而是包含完整的多阶段工作：

1. 编写 FF++ 子集下载脚本，控制下载范围与压缩级别；
2. 编写视频抽帧脚本，将公开视频序列转成图像级训练材料；
3. 编写数据集整理脚本，将多类伪造方式统一映射到标准监督学习目录；
4. 编写视觉模型训练脚本，实现迁移学习、验证评估与阈值冻结；
5. 编写离线特征导出脚本，将系统中间结果转化为融合模型可训练样本；
6. 编写融合模型训练脚本，完成结构化多特征学习；
7. 生成并整理权重文件与 metadata 文件；
8. 将训练后的模型资产接入后端运行时管理；
9. 保留启发式回退逻辑，保证模型不可用时系统仍能闭环；
10. 通过模型版本、阈值来源、可视化产物与结果页展示，将训练成果落实到完整系统中。

因此，这一部分工作体现的是数据工程、训练工程、部署工程与系统集成能力的结合，而不是单纯复现一个现成算法。
