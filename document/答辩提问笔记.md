# 答辩提问笔记

## 1. `feature_extraction_agent.py` 里调用的三个对象，在 `backends.py` 中到底怎么工作

先纠正一个表述：`FeatureExtractionAgent` 从 `backends.py` 引入的 `FaceDetectorManager`、`PublicFigureGallery`、`VisualBackendManager` 都不是“函数”，而是类。

在 `backend/app/agents/feature_extraction_agent.py` 中：

- `__init__()` 里实例化了三个对象  
  对应代码：`self.detector = FaceDetectorManager()`、`self.visual_backend = VisualBackendManager()`、`self.gallery = PublicFigureGallery()`
- 真正使用发生在 `run()` 里：
  - `self.detector.detect(image)`：做人脸检测
  - `self.visual_backend.analyze(face_crop, image.size, detected_face.bbox)`：做视觉特征提取和伪造分数计算
  - `self.gallery.match(detected_face.embedding or analysis["feature_embedding"])`：做人脸 embedding 与公共人物图库的匹配

也就是说，`FeatureExtractionAgent` 自己不直接写检测模型和视觉模型逻辑，而是把“检测”“视觉分析”“图库匹配”这三类能力分别委托给 `backends.py` 中的三个后端组件。

### 1.1 `FaceDetectorManager` 是怎么被调用的

`FaceDetectorManager` 定义在 `backend/app/services/backends.py` 中，它本质上是一个“检测器管理器”，不是一个具体模型。

它在初始化时会准备三个候选检测后端：

- `InsightFaceDetector()`
- `OpenCVHaarDetector()`
- `HeuristicFaceDetector()`

然后 `detect(image)` 会按顺序尝试：

1. 先调用 `InsightFaceDetector.detect(image)`
2. 如果没有结果，再调用 `OpenCVHaarDetector.detect(image)`
3. 还不行，再调用 `HeuristicFaceDetector.detect(image)`

只要某一个后端返回了有效人脸框，`FaceDetectorManager` 就会立刻返回，不再继续往下尝试。

所以在答辩时你可以直接说：

> `FeatureExtractionAgent` 本身不关心底层到底是 InsightFace、OpenCV 还是启发式检测器，它统一调用的是 `FaceDetectorManager.detect()`。真正的多后端选择逻辑被封装在 manager 里面。

### 1.2 `VisualBackendManager` 是怎么被调用的

`VisualBackendManager` 也是一个“视觉后端管理器”，它不是单一模型。

它内部管理两个视觉后端：

- `EfficientNetVisualBackend()`：真实视觉模型后端
- `HeuristicVisualBackend()`：启发式视觉后端

调用过程是这样的：

1. `FeatureExtractionAgent` 对每张人脸调用 `self.visual_backend.analyze(...)`
2. `VisualBackendManager.analyze(...)` 先尝试 `optional_backend.analyze(...)`
3. 如果真实模型权重不存在、metadata 缺失、模型加载失败，或者推理过程报错，就 `except` 后自动回退到 `fallback_backend.analyze(...)`

所以这部分的核心不是“只调一个模型”，而是“先尝试真实模型，失败就回退到启发式方案”。

它返回的结果不是只有一个真假标签，而是一组结构化视觉特征，包括：

- `deepfake_score`
- `quality_score`
- `dct_score`
- `blur_score`
- `compression_score`
- `face_size_ratio`
- `augmentation_consistency_score`
- `feature_embedding`
- `backend_name`

这些字段后面都会被封装进 `FaceResult`，再交给 `DecisionAgent` 继续处理。

### 1.3 `PublicFigureGallery` 是怎么被调用的

`PublicFigureGallery` 的作用不是做检测，而是做“人脸 embedding 与公共人物图库 embedding 的相似度匹配”。

调用过程是：

1. `FeatureExtractionAgent` 拿到当前人脸的 embedding  
   来源有两种：
   - 如果检测器本身提供 embedding，就优先用 `detected_face.embedding`
   - 如果没有，就退回到视觉分析阶段生成的 `analysis["feature_embedding"]`
2. 然后调用 `self.gallery.match(...)`
3. `PublicFigureGallery.match()` 会先读取 `gallery.json`
4. 再把当前 embedding 和图库中每个人物的 reference embedding 做余弦相似度计算
5. 如果相似度超过阈值，就把这个人物加入候选列表

它最后返回的是一个名字列表，比如“某公众人物候选”，这个列表会写入 `FaceResult.public_figure_candidates`。

这一步的意义在于：

- 它不直接判定真假
- 它给 `DecisionAgent` 提供“是否涉及公共人物语境”的辅助证据
- 一旦匹配到公共人物，后面的策略选择更容易走 `visual_plus_semantic`

所以答辩时你可以说：

> `PublicFigureGallery` 的作用不是做人脸识别登录那种身份确认，而是给 deepfake 决策提供“公共人物候选”上下文，用于触发更谨慎的策略选择。

## 2. `InsightFaceDetector` 和 `OpenCVHaarDetector` 是怎么定义的

### 2.1 `InsightFaceDetector` 的定义思路

`InsightFaceDetector` 是一个类封装，不是在初始化时就立刻把模型加载进来，而是采用“延迟加载”的方式。

它的结构是：

- `__init__()`：只定义一些状态字段，比如 `_backend`、`_available`、`backend_name`、`provider_name`
- `is_available()`：真正检查依赖、模型缓存和运行 provider，并在这里尝试导入第三方库、初始化 `FaceAnalysis`
- `detect(image)`：只有在 `is_available()` 返回真时才执行真实检测逻辑

也就是说，`InsightFaceDetector` 不是“构造时加载模型”，而是“第一次真正需要用的时候再检查并加载”。

这样设计的好处有两个：

1. 如果机器上没有装 `insightface` 或 `onnxruntime`，系统启动不会直接崩
2. 即使没有下载好 `buffalo_l` 模型缓存，系统也还能继续回退到 OpenCV 或启发式检测器

### 2.2 `OpenCVHaarDetector` 的定义思路

`OpenCVHaarDetector` 也是同样的思路：

- `__init__()` 先把 `_cascade` 和 `_available` 设为初始状态
- `is_available()` 里才去尝试 `import cv2`
- 如果导入成功，就去加载 OpenCV 自带的人脸 Haar Cascade XML 文件
- `detect(image)` 里才真正做人脸框检测

这里也不是在模块顶层直接初始化 OpenCV 检测器，而是按需加载。

它的检测逻辑比 InsightFace 简单很多：

- 先把图片转灰度
- 必要时缩放大图
- 用 Haar Cascade 做人脸框搜索
- 把结果封装成 `DetectedFace`

因此它在项目里本质上是“第二优先级回退检测器”。

## 3. 为什么这两个类没有在 `backends.py` 顶部直接 `import` 对应 Python 库

这是一个很适合答辩回答的工程设计点。

原因不是漏写 import，而是**故意做成延迟导入（lazy import）**。

### 3.1 直接顶部导入会带来的问题

如果在 `backends.py` 文件顶部直接写：

- `import insightface`
- `import onnxruntime`
- `import cv2`

那么只要运行环境里缺少任何一个依赖，整个后端模块在 import 阶段就会直接报错，项目甚至启动不起来。

这和本项目“要允许模型缺失时回退运行”的目标是冲突的。

### 3.2 当前写法的实际作用

现在的写法是把 import 放进 `is_available()` 或相关方法内部，比如：

- `InsightFaceDetector.is_available()` 里才 `import onnxruntime` 和 `from insightface.app import FaceAnalysis`
- `OpenCVHaarDetector.is_available()` 里才 `import cv2`

这样做的效果是：

1. 后端启动时不会因为检测器依赖缺失而整体崩掉
2. 某个后端不可用时，只会把 `_available` 设为 `False`
3. 上层 `FaceDetectorManager` 会自动继续尝试下一个可用后端

所以你明天如果被问到“为什么这里没直接 import 第三方库”，最稳的回答就是：

> 这是故意做的懒加载设计。因为项目要求在真实模型或依赖不完整时仍然可以运行，所以把第三方库导入放到 `is_available()` 里，只有真正需要启用这个后端时才尝试导入。这样即使 InsightFace 或 OpenCV 不可用，系统也不会启动失败，而是自动回退到其他检测后端。

## 4. 这道题最推荐的现场回答版本

如果老师现场直接问你这一题，你可以用下面这段回答：

> 在 `FeatureExtractionAgent` 里，我没有把检测、视觉分析和公共人物匹配直接写死，而是分别委托给 `FaceDetectorManager`、`VisualBackendManager` 和 `PublicFigureGallery`。`FaceDetectorManager` 内部按顺序尝试 InsightFace、OpenCV Haar 和启发式检测器；`VisualBackendManager` 先尝试 EfficientNet 视觉模型，失败再回退到 heuristic visual；`PublicFigureGallery` 负责把当前人脸 embedding 和图库 embedding 做相似度匹配，给后续决策提供公共人物上下文。至于 `InsightFaceDetector` 和 `OpenCVHaarDetector` 没有在文件顶部直接 import 第三方库，是因为这里做了懒加载设计：只有真正调用这个检测后端时才在 `is_available()` 里导入依赖。这样即使某些库没装好或者模型资产没到位，整个系统也不会启动失败，而是自动回退到下一级后端。  

## 5. 为什么 `DecisionAgent` 不直接输出真假，而是先做策略选择

`DecisionAgent` 的设计重点不是“直接分类”，而是“先判断当前样本应该走哪条判定路径”，然后再输出最终标签。

它这么做的原因是：这个项目面对的输入并不总是标准、清晰、单人脸、质量良好的样本。实际会出现：

- 无人脸
- 多人脸
- 图像质量低
- 分数落入灰区
- 公共人物候选
- 视觉指标之间互相冲突

如果直接拿一个分数强行二分类，系统会很不稳，也不利于解释。

所以 `DecisionAgent` 的工作顺序是：

1. 先根据 `FeatureExtractionAgent` 的输出构建 `TaskProfile`
2. 再根据画像判断这个任务适合：
   - `visual_only`
   - `visual_plus_semantic`
   - `review_only`
3. 最后才进入融合判定和结果输出

这样做的意义是：

- 简单样本走简单路径，效率高
- 复杂样本触发更谨慎的分析
- 不适合判定的样本直接转人工复核

所以现场回答时不要说“DecisionAgent 就是另一个分类器”，更准确的说法是：

> `DecisionAgent` 是任务级决策器，它先决定这个样本该怎么判，再去给出真假或复核结论。

## 6. `evidence_chain` 和 `thinking_chain` 有什么区别

这两个概念很容易被混在一起，但你答辩时一定要区分开。

### 6.1 `evidence_chain`

`evidence_chain` 是系统正式返回的结构化证据链，它是一个数组，里面每一项都是 `EvidenceEntry`。

每条证据项都包含：

- `step`
- `actor`
- `timestamp`
- `summary`
- `details`

它的定位是“审计记录”。

也就是说，即使不用大模型，这个系统也照样可以返回证据链，因为证据链本质上是系统处理过程的结构化记录。

### 6.2 `thinking_chain`

`thinking_chain` 不是单独的一条总链，而是某些证据项内部可能附带的一个字段，通常出现在：

- `strategy_selection`
- `semantic_analysis`

它的来源是大模型返回的详细推理说明。

所以更准确的关系是：

- `evidence_chain` 是主容器
- `thinking_chain` 是其中某些步骤的补充说明字段

### 6.3 为什么项目对外强调证据链而不是纯 CoT

因为结构化证据链更稳定、可控、可审计。  
如果完全依赖开放式 CoT，会出现输出不稳定、格式不统一、难以约束的问题。

所以答辩时建议你这样说：

> 我们正式对外返回的是结构化 `evidence_chain`，而不是把大模型原始思维过程直接当成系统主输出。`thinking_chain` 只是某些步骤中的补充字段，用来增强解释性，但系统的正式审计基础仍然是证据链。

## 7. 为什么要单独做 `model_registry.py`

`model_registry.py` 的作用不是“多写一个文件”，而是把模型状态管理和阈值管理统一集中起来。

它主要做两件事：

### 7.1 管理模型当前是否可用

它会统一检查：

- detector 是否 ready
- visual 权重和 metadata 是否存在
- fusion 模型和 metadata 是否存在
- semantic 模型是否已配置
- LLM 是否已配置 API key

这样前端和后端别的模块就不需要各自重复判断模型状态。

### 7.2 统一管理阈值来源

它会按优先级读取阈值：

1. 融合模型 metadata
2. 视觉模型 metadata
3. 默认配置

这意味着阈值逻辑不会散落在各个业务文件中，而是集中由 `ModelRegistry` 决定。

### 7.3 为什么这么设计更好

如果没有 `model_registry.py`，那后端很多地方都要自己判断：

- 模型文件是否存在
- metadata 是否存在
- 当前应该显示哪个后端
- 当前阈值应该从哪里取

这样代码会很乱，也不利于维护。

所以答辩时可以这样说：

> `model_registry.py` 相当于系统的模型配置与运行状态中心，把“模型是否就绪”和“阈值从哪里来”统一管理，减少业务模块之间的重复判断。

## 8. 为什么 `repository.py` 和 `database.py` 要分层

这个问题很适合体现你对代码结构的理解。

最简单的说法是：

- `database.py` 负责“直接操作 SQLite”
- `repository.py` 负责“把数据库记录转换成业务对象”

### 8.1 `database.py` 干的事情

`database.py` 更底层，它负责：

- 建表
- 插入任务
- 更新任务
- 保存结果 JSON
- 查询单条任务
- 查询任务列表

它处理的是数据库字段和 SQL 语句。

### 8.2 `repository.py` 干的事情

`repository.py` 是业务层封装，它负责：

- 调用 `database` 获取原始记录
- 把数据库记录转换成 `TaskStatusResponse`
- 把 `result_json` 反序列化成 `DetectionResult`
- 把任务列表转换成 `TaskSummaryResponse`

也就是说，它处理的是“系统业务对象”，而不是原始 SQL。

### 8.3 为什么不直接把这些都写在一个文件里

如果全写在一起，会导致：

- SQL 逻辑和业务逻辑混杂
- API 层直接依赖数据库细节
- 后续修改数据库结构时影响面太大

现在这种分层好处是：

- `database.py` 只关心存取
- `repository.py` 只关心业务对象
- 上层路由和调度器不需要直接碰 SQL

答辩时你可以这样回答：

> `database.py` 是底层数据访问层，`repository.py` 是面向业务的仓储层。这样分层后，路由和调度器操作的是任务对象和结果对象，而不是直接写 SQL，结构更清晰，也更方便维护。

## 9. 为什么结果页里要显示 `model_versions`

这个问题如果答得好，会显得你对系统可解释性理解很到位。

`model_versions` 的作用不是装饰信息，而是告诉使用者：

- 当前到底是哪个 detector 在工作
- 当前用的是视觉模型还是 heuristic visual
- 当前融合是 XGBoost 还是 heuristic fusion
- 当前语义是 LLM、本地模型还是规则后端
- 当前阈值来自 metadata 还是默认配置

这意味着系统不是只给一个“真假标签”，而是把“这个标签是由哪套后端产生的”也一起展示出来。

这对于答辩特别重要，因为老师很容易问：

- 你现在演示的是不是实模型？
- 这次结果到底是谁给出来的？
- 当前阈值是写死的还是训练出来的？

`model_versions` 就是专门解决这些追问的。

所以你可以这样总结：

> `model_versions` 的本质作用是让结果可追溯。用户看到的不只是结论，还能看到当前到底是哪套后端和哪套阈值在工作。

## 10. 为什么系统不用“一个大模型全做完”

这个问题本质上是在考你有没有工程判断。

项目没有设计成“上传图片后交给一个大模型直接输出真假”，原因主要有三点：

### 10.1 可控性不够

如果全交给一个大模型自由输出，就很难保证：

- 输出格式稳定
- 策略边界清晰
- 阈值和规则一致

现在的系统把大模型限制在“策略选择”和“语义辅助”两个环节，核心判定逻辑仍然受规则、阈值和融合模型约束。

### 10.2 可解释性不够稳定

纯大模型输出很容易文本很多，但结构不稳定。  
而本项目更强调结构化证据链，所以大模型只是辅助，不是主宰。

### 10.3 可用性风险太高

如果模型服务不可用，系统就会整体失效。  
现在的设计有完整回退链路，即使大模型不可用，系统仍然能继续工作。

所以你可以这样回答：

> 本项目没有走“一个大模型包办一切”的路线，而是把大模型约束在局部环节，用传统视觉后端、融合模型、规则判定和回退机制保证系统整体稳定性。这更符合工程系统设计，而不是纯演示式方案。
