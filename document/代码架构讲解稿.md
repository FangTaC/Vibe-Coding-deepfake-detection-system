# 基于智能体协作的 Deepfake 人脸图像检测系统

## 代码架构讲解稿

说明：本文稿基于当前仓库代码、运行配置和现有模型接入状态整理，适合直接用于答辩讲解、老师提问应答和 Word 文档整理。部分表述属于根据现有实现做出的合理归纳，重点强调系统工程闭环、智能体协作思路和模型接入设计，而不把当前演示版本表述成“最终实验指标版本”。

## 一、整体架构总览

这个项目可以概括为“前端上传入口 + 后端异步任务调度 + 双智能体协同分析 + 多后端融合判定 + 结构化结果展示”。

从系统链路上看，用户先在前端上传一张图片，前端调用后端接口创建检测任务。后端创建任务后，不是同步阻塞返回，而是把任务交给异步调度器执行。任务执行时，先由特征提取智能体完成人脸检测、裁剪、视觉分数计算和可视化产物生成；再由决策智能体根据任务画像选择检测策略，决定是只走视觉路径、走视觉加语义联合路径，还是直接转人工复核。最终系统输出图像级结论、置信度、融合分数、复核原因、证据链、模型信息以及热力图等产物，再由前端统一展示。

如果老师问“这个系统和普通分类器有什么区别”，建议你回答：本项目不是单纯做一个二分类模型，而是把 Deepfake 检测做成一个可运行、可解释、可回退、可审计的完整系统。它的核心不只是识别真假，还包括任务调度、证据组织、异常处理和人工复核机制。

## 二、前端架构与页面功能

前端采用 Vue3 + Vite 实现，入口页面集中在 `frontend/src/App.vue`。页面整体分为左右两栏：

- 左侧是上传与历史任务区。
- 右侧是结果详情区和证据链时间线区。

### 1. 前端页面主控逻辑

`App.vue` 是整个前端的总控组件，主要负责四件事：

- 初始化时调用系统状态接口，判断 LLM 是否可用，并把状态显示在首页顶部。
- 调用历史任务列表接口，加载最近任务。
- 处理上传动作，创建新任务。
- 轮询任务状态，任务完成后再拉取完整结果。

也就是说，前端不是一次上传完就等结果，而是走“创建任务 -> 查询状态 -> 完成后拉结果”的异步任务模式，这样更贴合真实系统流程，也更适合展示多阶段执行过程。

### 2. 上传区

上传区域对应 `frontend/src/components/UploadPanel.vue`。它支持两种输入方式：

- 用户本地选择图片文件。
- 点击示例图片，前端自动把示例图转成 File 对象后提交。

这个组件本身只做输入和基础校验，比如没有选中文件时阻止提交；真正的格式和大小校验在后端完成。这样设计的好处是前端体验更友好，后端规则也更统一。

### 3. 历史任务区

历史任务由 `frontend/src/components/TaskHistory.vue` 展示。它会列出最近任务的文件名、当前阶段、状态标签和部分置信度信息。老师如果问为什么要做历史任务页，你可以说：因为这不是一次性的单页面识别，而是面向“任务”建模，既方便演示，也方便后续做追溯、复查和多样本对比。

### 4. 结果展示区

结果页面由 `frontend/src/components/ResultPanel.vue` 渲染，核心展示内容包括：

- 当前任务状态、阶段与进度条。
- 图像级结论、融合分数、置信度。
- 视觉摘要、语义摘要、复核原因。
- 原图标注图。
- 每张人脸的人脸裁剪图与热力图。
- 每张人脸的视觉分数、质量分数、DCT 分数、一致性分数。
- 公共人物候选信息。
- 模型信息与阈值来源。
- 如果启用了 LLM，还会显示策略选择时的推理链路。

### 5. 证据链时间线区

证据链由 `frontend/src/components/EvidenceTimeline.vue` 展示。它把后端返回的 `evidence_chain` 数组按照时间线显示出来，包含每一步的：

- step
- actor
- timestamp
- summary
- details

如果某一步包含 `thinking_chain`，前端会单独高亮，并支持展开查看大模型推理过程。这一点是本项目答辩里的亮点，因为它体现的是“结构化可审计输出”，而不是单纯给一个标签。

## 三、上传图片功能的完整实现链路

老师很可能会问“上传一张图片之后，代码到底怎么走”。你可以按下面这条主线回答：

第一步，前端在 `frontend/src/api/client.js` 中调用 `POST /api/tasks` 接口，把图片封装成 `FormData` 上传。

第二步，后端 `backend/app/api/routes.py` 中的 `create_task` 路由接收 `UploadFile`，读取二进制内容后交给 `DetectionOrchestrator.create_task()`。

第三步，调度器在 `backend/app/services/pipeline.py` 中先校验扩展名和大小。当前只允许 `.jpg`、`.jpeg`、`.png`、`.webp`，文件大小默认不超过 10MB。

第四步，上传文件通过 `storage_service.save_upload()` 保存到 `backend/storage/uploads/<task_id>/` 目录下，同时任务基本信息写入 SQLite 数据库。

第五步，调度器调用 `submit_task()`，把这个任务交给线程池异步执行，而不是在接口里同步跑完整个检测流程。

第六步，前端收到任务 ID 后，先查一次状态接口 `GET /api/tasks/{task_id}`，再通过轮询持续查看任务有没有完成。

第七步，任务执行完后，前端调用 `GET /api/tasks/{task_id}/result`，拿到完整检测结果并渲染页面。

这条链路说明了本系统是“任务驱动”的，不是一个简单的同步接口分类器。

## 四、后端 API 路由与任务调度流程

### 1. 后端入口

后端主入口在 `backend/app/main.py`，使用 FastAPI 构建应用，并注册跨域中间件和路由。路由统一挂在 `/api` 前缀下。

### 2. API 路由

核心路由集中在 `backend/app/api/routes.py`：

- `GET /api/health`：健康检查。
- `GET /api/status`：返回系统运行时状态，包括 detector、visual、fusion、semantic、llm 是否就绪。
- `GET /api/tasks`：返回任务历史列表。
- `POST /api/tasks`：上传图片并创建任务。
- `GET /api/tasks/{task_id}`：查询任务执行状态。
- `GET /api/tasks/{task_id}/result`：获取最终检测结果。
- `GET /api/tasks/{task_id}/artifacts/{name}`：访问可视化产物，如原图标注图、对齐人脸图、热力图。

### 3. 任务调度器

任务调度器定义在 `backend/app/services/pipeline.py` 中，类名为 `DetectionOrchestrator`。它是后端主流程的核心。

它主要做三件事：

- 创建任务并写入数据库。
- 使用 `ThreadPoolExecutor` 异步执行检测任务。
- 在任务执行过程中持续更新状态、阶段、进度和当前智能体。

当前阶段主要有：

- `queued`
- `preprocessing`
- `feature_extraction`
- `decision`
- `completed`
- `failed`

老师如果问“为什么要专门做调度器”，建议你回答：因为项目要展示的不只是单个模型推理，而是完整任务生命周期管理。调度器把上传、执行、状态更新、失败处理和结果落库统一管理，这样系统才具备可用性和可扩展性。

### 4. 数据库存储

数据库在 `backend/app/db/database.py` 中实现，使用 SQLite。`tasks` 表里保存：

- 任务 ID
- 文件名
- 内容类型
- 当前状态
- 当前阶段
- 进度
- 当前智能体
- 输入文件路径
- 最终结果 JSON
- 错误信息
- 创建与更新时间

这意味着系统的“任务状态页”和“历史任务页”都不是前端自己记的，而是后端数据库驱动的。

## 五、智能体框架是怎么实现的

本项目的智能体并不是一个通用 Agent 平台，而是面向检测流程拆分出来的“角色化智能体”。

从实现上看，主要有两个核心智能体：

- `FeatureExtractionAgent`
- `DecisionAgent`

这两个智能体由调度器顺序调用，形成一个“先取证、后决策”的处理链。

### 1. 特征提取智能体做什么

`backend/app/agents/feature_extraction_agent.py` 中的 `FeatureExtractionAgent` 主要负责“把原始图片变成可供决策的结构化证据”。它的职责包括：

- 读取输入图像。
- 调用人脸检测后端得到人脸框。
- 对每张人脸裁剪、对齐。
- 调用视觉后端得到伪造分数、质量分数、DCT 分数、模糊分数、压缩分数、一致性分数。
- 计算公共人物候选。
- 生成原图标注图、人脸裁剪图、热力图等可视化产物。
- 生成前半段证据链。

它的输出不是最后结论，而是一个特征包，里面含有：

- `faces`
- `artifacts`
- `evidence`
- `input_summary`
- `model_versions`

你在答辩时可以把它解释成“取证智能体”，因为它的核心任务是收集证据，而不是直接拍板。

### 2. 决策智能体做什么

`backend/app/agents/decision_agent.py` 中的 `DecisionAgent` 主要负责“根据已有证据做受限决策”。它的职责包括：

- 根据所有人脸结果构建任务画像 `TaskProfile`。
- 选择策略：`visual_only`、`visual_plus_semantic` 或 `review_only`。
- 必要时触发语义分析。
- 调用融合后端生成图像级融合分数。
- 根据阈值与规则输出最终标签。
- 给出复核原因、视觉摘要、语义摘要、模型版本信息。
- 把整个决策过程整理为结构化证据链并返回。

这里的关键不是“让大模型自己自由发挥”，而是“让决策智能体只能在预注册策略中选一种”。这能体现系统设计上的可控性。

### 3. 为什么说这是“智能体协作”

因为这两个智能体不是重复干活，而是分工明确：

- 特征提取智能体负责“感知和取证”。
- 决策智能体负责“策略选择和结论生成”。

前者输出结构化特征，后者基于这些特征再选择合适路径完成判定。这个结构和普通单模型直接输出标签相比，更容易扩展，也更适合做证据链展示。

## 六、策略选择、语义分析与证据链生成

### 1. 任务画像如何构建

决策智能体会先根据视觉结果生成任务画像 `TaskProfile`，主要字段包括：

- 人脸数量
- 最大人脸分数
- 平均人脸分数
- 平均质量分数
- 公共人物候选
- 是否需要语义检查
- 视觉冲突程度 `conflict_level`

是否需要语义分析，不是随机决定的，而是由规则触发，比如：

- 分数落入灰区
- 多人脸场景
- 有公共人物候选
- 视觉指标之间存在较强冲突

### 2. 策略是怎么选的

策略选择逻辑在 `DecisionAgent._select_strategy()` 中实现。当前优先尝试使用 `QwenStrategyBackend` 做策略选择；如果 LLM 不可用或返回异常，就回退到规则式策略选择。

可选策略只有三种：

- `visual_only`
- `visual_plus_semantic`
- `review_only`

这三种策略的意义分别是：

- 简单明确样本直接走视觉判断。
- 复杂样本走视觉加语义联合判断。
- 不适用样本直接进入人工复核。

答辩时你可以强调：这里用 LLM 的方式不是“让它替代判定”，而是“让它在受约束的策略空间内做策略选择，并输出理由”。

### 3. 语义分析怎么触发

当策略选择结果为 `visual_plus_semantic` 时，系统才会调用语义分析后端。

语义分析后端在 `backend/app/services/backends.py` 中由 `SemanticBackendManager` 管理，优先级是：

- 通义千问视觉 API 后端
- 本地 Qwen3-VL
- 规则式语义后端

语义分析输出的不是最终真假标签，而是：

- `semantic_flags`
- `semantic_score`
- `summary`
- 可选 `thinking_chain`

这说明语义模块扮演的是“辅助证据来源”，而不是最终裁判。

### 4. 证据链如何生成并返回

证据链由两个智能体分段生成，最后在结果对象中统一返回。

特征提取智能体会写入：

- `input_loaded`
- `face_detection`
- `feature_extraction`

决策智能体会继续写入：

- `strategy_selection`
- `semantic_analysis`
- `final_decision`

每个证据项都包含：

- 执行步骤
- 执行者
- 时间戳
- 摘要
- 详细字段

如果 LLM 有参与，详细字段中还会带上 `thinking_chain`。前端拿到后，一部分在结果页展示，一部分在时间线组件中展示。

## 七、结果页各模块与后端实现的对应关系

老师很可能会指着前端页面问“这个模块后端对应哪里”。你可以按下面的映射回答。

### 1. 状态区和进度条

前端状态区来自 `GET /api/tasks/{task_id}` 返回的 `TaskStatusResponse`，字段包括：

- `status`
- `stage`
- `progress`
- `current_agent`

这些字段由调度器在 `pipeline.py` 中逐阶段更新。

### 2. 图像级标签、融合分数、置信度、复核原因

这些数据来自 `DetectionResult`，由 `DecisionAgent.run()` 生成，其中：

- 标签来自 `_finalize_label()`
- 融合分数来自 `_fuse_scores()`
- 置信度来自 `_compute_confidence()`
- 复核原因来自最终标签决策逻辑

### 3. 原图标注图

来自特征提取智能体中的 `save_annotated_image()`，文件保存在 `storage/artifacts/<task_id>/annotated_original.jpg`，前端通过产物 URL 访问。

### 4. 人脸裁剪图和热力图

分别由：

- `save_face_crop()`
- `save_heatmap()`

生成后保存在任务产物目录中，再通过 `artifacts` 数组返回给前端。

### 5. 证据链时间线

直接来自 `DetectionResult.evidence_chain`。前端不自己拼逻辑，而是按照后端给的结构化记录渲染。

### 6. 模型信息

结果页中的“检测后端、视觉后端、融合后端、语义后端、阈值来源、模型名”等信息来自 `DetectionResult.model_versions`。这部分由：

- 特征提取智能体补充 detector 和 visual 相关信息。
- 决策智能体补充 fusion、semantic、threshold 相关信息。

这也是你答辩时可以突出的一点：系统不仅给结果，还告诉你“是哪个后端给出的结果”。

## 八、模型权重、metadata、配置文件分别起什么作用

### 1. 配置文件

配置集中在 `backend/app/core/config.py`。主要定义：

- API 前缀
- 允许上传的格式与大小
- 默认阈值
- 最大人脸数
- 线程池 worker 数
- 模型权重路径
- metadata 路径
- LLM 与语义模型配置
- 存储目录位置

配置文件的作用是把“系统参数”和“代码逻辑”分离，便于部署和切换。

### 2. 模型权重文件

当前设计中主要有两个核心模型资产：

- `efficientnet_b0_deepfake.pt`
- `xgboost_fusion.json`

它们分别对应视觉模型和融合模型本体。没有权重文件，系统不能启用真实模型后端。

### 3. metadata 文件

metadata 文件不是可有可无的补充说明，而是运行时契约的一部分。当前至少包括：

- 模型名称
- 生成时间
- 数据集版本
- 指标
- 阈值
- 特征键

后端的阈值读取会优先从 metadata 中获取，而不是写死在代码里。这样做的意义是：训练阶段和部署阶段可以通过 metadata 对接起来，系统知道当前模型对应哪套阈值和哪个数据版本。

### 4. 模型注册器

`backend/app/services/model_registry.py` 负责统一管理模型状态与阈值策略。它主要做两件事：

- 判断 visual、fusion、semantic 等后端当前是否真正 ready。
- 决定阈值的读取来源。

阈值优先级是：

- 融合模型 metadata
- 视觉模型 metadata
- 代码默认阈值 0.30 / 0.70

所以如果老师问“阈值是不是大模型动态改的”，你可以明确回答：不是。阈值只来自 metadata 或默认配置，不允许由 LLM 动态修改。

## 九、回退机制是如何工作的

回退机制是本项目一个非常重要的工程亮点。

### 1. 人脸检测回退

`FaceDetectorManager` 的优先顺序是：

- InsightFace
- OpenCV Haar
- HeuristicFaceDetector

也就是说，如果高精度检测器资产没准备好，系统仍然不会整体失效，而是继续走较弱但可用的后端。

### 2. 视觉后端回退

`VisualBackendManager` 优先尝试 EfficientNet-B0；如果权重文件或 metadata 缺失，就自动回退到 `heuristic_visual`。

### 3. 融合后端回退

`FusionBackendManager` 优先尝试 XGBoost；如果模型不可用，则回退到 `heuristic_fusion`。

### 4. 语义后端回退

语义分析优先尝试 API 模型，再尝试本地模型，最后回退到 `rule_based_semantic`。如果语义分析超时，还会加上 `SEMANTIC_TIMEOUT` 标记，并主动转人工复核。

### 5. LLM 策略选择回退

策略选择优先用 `QwenStrategyBackend`，失败就回退到规则策略。也就是说，LLM 不可用时不会影响系统主流程，只会让系统从“LLM 参与策略说明”退化为“规则式策略说明”。

### 6. 为什么要做这么多回退

因为项目答辩和实际系统演示都不能接受“某个模型没准备好就全系统崩掉”。本项目强调的是“可用性优先、闭环优先、可审计优先”。所以即使真实模型资产未全部到位，系统也要能继续跑通上传、检测、展示、证据链和复核流程。

## **关键内容代码实现讲解**

这一部分建议你在老师开始追问“具体怎么写的”时使用。和前面偏架构层的讲法不同，这一节是直接对着代码文件和行号去讲，适合说明“不是只会讲概念，而是真的把系统完整写出来了”。

### 1. 双 Agent 异步调度到底是怎么实现的

双 Agent 异步调度的核心入口在 `backend/app/services/pipeline.py:18-84`。

第一层是调度器类 `DetectionOrchestrator`，它在 `pipeline.py:18-24` 初始化了一个 `ThreadPoolExecutor` 和 `futures` 字典。这里的线程池不是为了并发执行两个 Agent，而是为了把整个检测任务异步提交出去，让 Web 接口先返回任务 ID，再由后台线程继续跑完整流程。

第二层是任务创建逻辑，在 `pipeline.py:33-38`。这里完成三件事：

- 调用 `validate_upload()` 校验扩展名和大小。
- 生成唯一的 `task_id`。
- 把上传图片保存到本地存储，并把任务信息写入数据库。

第三层是真正的异步提交逻辑，在 `pipeline.py:77-81`。`submit_task()` 会先检查这个任务是否已经在跑，如果没有，就用 `self.executor.submit(self.run_task, task_id)` 提交后台任务。也就是说，前端上传成功后不会卡在接口里等完整检测，而是进入任务轮询模式。

第四层是双 Agent 的串行协作执行逻辑，在 `pipeline.py:40-75`。这里能清楚看到：

- `pipeline.py:45` 实例化 `FeatureExtractionAgent`
- `pipeline.py:46` 实例化 `DecisionAgent`
- `pipeline.py:48-61` 先把任务状态更新到 `preprocessing` 和 `feature_extraction`
- `pipeline.py:62` 调用 `feature_agent.run(task_id, image_path)`
- `pipeline.py:63-69` 再把任务状态切换到 `decision`
- `pipeline.py:70` 调用 `decision_agent.run(task_id, image_path, features)`
- `pipeline.py:71` 把最终结果保存进数据库

这就是“异步调度 + 双 Agent 协作”的关键实现。异步体现在整个任务是由线程池后台执行；双 Agent 体现在单个任务内部先调用 `feature_agent`，再调用 `decision_agent`，而且每个阶段都会把 `current_agent` 和 `progress` 写回数据库。

如果老师问“为什么说是双 Agent 异步调度，而不是普通函数调用”，建议你这样回答：

第一，系统外层是异步任务调度，不是同步接口阻塞；第二，任务内层不是一个函数做完所有事，而是两个有明确职责边界的 Agent 依次接管任务；第三，状态、阶段和当前执行智能体都被持久化，所以前端能实时看到 Agent 切换过程。

### 2. 上传后如何把异步任务链路真正打通到前端

后端路由侧：

### 3. 特征提取 Agent 是如何接管任务的

特征提取 Agent 的实现集中在 `backend/app/agents/feature_extraction_agent.py:17-189`。

最关键的逻辑在 `feature_extraction_agent.py:25-189` 的 `run()` 方法。

第一步，在 `feature_extraction_agent.py:26-28` 读取图像并调用人脸检测器。

第二步，在 `feature_extraction_agent.py:28-40` 先写入第一条证据链 `input_loaded`，说明图像尺寸、模式等输入信息。这意味着证据链从任务一开始就建立了，而不是最后临时拼接。

第三步，如果没检测到人脸，`feature_extraction_agent.py:42-83` 会直接生成：

- `face_detection` 证据项
- `feature_extraction` 证据项
- 原图标注图产物
- 一个没有 `faces` 的特征包

这一步很重要，因为它说明系统不是“无人脸直接报错”，而是把“无人脸”也当成一种可解释结果返回给决策 Agent。

第四步，如果检测到人脸，`feature_extraction_agent.py:85-144` 会遍历每张人脸，分别完成：

- 裁剪人脸 `crop_face()`
- 保存人脸图 `save_face_crop()`，对应 `feature_extraction_agent.py:107-108`
- 保存热力图 `save_heatmap()`，对应 `feature_extraction_agent.py:109`
- 调用视觉后端分析 `self.visual_backend.analyze()`，对应 `feature_extraction_agent.py:110`
- 计算公共人物候选 `self.gallery.match()`，对应 `feature_extraction_agent.py:111`
- 把所有指标封装到 `FaceResult`，对应 `feature_extraction_agent.py:126-143`

第五步，在 `feature_extraction_agent.py:146-158` 写入 `feature_extraction` 证据项，说明已经完成多少张人脸的视觉证据提取。

第六步，在 `feature_extraction_agent.py:172-188` 返回完整特征包，包含：

- `faces`
- `artifacts`
- `evidence`
- `input_summary`
- `model_versions`

所以你可以把这部分概括成：特征提取 Agent 负责把“原始图片”加工成“可供决策 Agent 消费的结构化证据对象”。

### 4. 决策 Agent 如何调用大模型并形成 CoT

这一部分是老师最容易追问的亮点，核心代码在 `backend/app/agents/decision_agent.py:35-173` 和 `backend/app/services/backends.py:634-779`。

#### 4.1 决策 Agent 如何接收上一个 Agent 的输出

在 `decision_agent.py:35-46`，`DecisionAgent.run()` 直接接收 `features` 参数，也就是特征提取 Agent 的输出。这里先取出：

- `faces`
- `evidence`
- `artifacts`
- `input_summary`
- `model_versions`

这说明第二个 Agent 不是重新处理原图，而是消费第一个 Agent 已经整理好的结构化中间结果。

#### 4.2 任务画像如何生成

在 `decision_agent.py:58` 调用 `_build_task_profile()`，具体逻辑在 `decision_agent.py:256-282`。这里把多张人脸结果压缩成任务级画像，核心字段有：

- `face_count`
- `max_face_score`
- `mean_face_score`
- `quality_score`
- `public_figure_candidates`
- `need_semantic_check`
- `conflict_level`

这一步很关键，因为后续 LLM 做策略选择时，吃的不是原图，而是这个任务画像。

#### 4.3 大模型策略选择是怎么调的

在 `decision_agent.py:59` 调用 `_select_strategy()`，实现位于 `decision_agent.py:284-300`。

这里的逻辑是：

- `decision_agent.py:288-292`：如果 `QwenStrategyBackend` 可用，就调用它，让大模型返回策略 JSON
- `decision_agent.py:293-296`：如果大模型报错，就回退到规则策略
- `decision_agent.py:297-300`：如果大模型本身不可用，也直接走规则策略

真正的大模型接口调用写在 `backend/app/services/backends.py:634-697` 的 `QwenStrategyBackend` 中：

- `backends.py:639-652` 是系统提示词，明确要求模型只能在 `visual_only / visual_plus_semantic / review_only` 三个策略中选，并且必须输出 `thinking_chain`
- `backends.py:667-697` 是 `select_strategy()` 方法
- `backends.py:678-691` 用 `httpx.post()` 调用通义千问兼容接口
- `backends.py:693-696` 解析返回 JSON，如果缺少 `thinking_chain`，就直接把模型原始文本填进去

所以答辩时你可以明确说：CoT 不是前端自己生成的，也不是写死在页面上的，而是由决策 Agent 在服务端调用 LLM 后，把 `thinking_chain` 放进策略选择结果，再统一写入证据链。

#### 4.4 CoT 是怎么进入证据链的

这一步在 `decision_agent.py:60-82`。

具体过程是：

- `decision_agent.py:60-66` 先构造 `strategy_detail`
- `decision_agent.py:67-68` 如果有 `strategy_decision.thinking_chain`，就把它塞进 `strategy_detail["thinking_chain"]`
- `decision_agent.py:71-82` 再把这整个对象作为 `EvidenceEntry(step="strategy_selection")` 写入证据链

这一步很重要，因为它说明 CoT 不是单独走另一条接口，而是正式成为证据链中的一部分。

#### 4.5 语义分析阶段的 CoT 是怎么来的

如果策略要求语义分析，在 `decision_agent.py:88-112` 会继续调用 `semantic_backend.analyze()`。

语义后端管理器在 `backends.py:595-631`：

- `backends.py:601-608` 优先尝试调用通义千问视觉 API
- `backends.py:613-616` 次选本地 Qwen3-VL
- `backends.py:617-631` 失败或超时则回退到规则式语义后端

其中，通义千问视觉后端在 `backends.py:700-779`：

- `backends.py:705-718` 规定返回 JSON 中必须包含 `thinking_chain`
- `backends.py:747-778` 把图片转成 base64 之后发给多模态模型
- `backends.py:776-778` 如果 JSON 里没有 `thinking_chain`，就把原始返回文本补进去

返回到 `DecisionAgent` 后，`decision_agent.py:99-100` 会把语义 `thinking_chain` 加入 `semantic_detail`，`decision_agent.py:101-112` 再把它写入 `semantic_analysis` 这条证据项。

### 5. 前端上的大模型 CoT 全流程是怎么实现的

这部分前后端是完整闭环，对应代码主要在：

- 后端：`backend/app/agents/decision_agent.py:59-112`
- 前端结果页：`frontend/src/components/ResultPanel.vue:35-45, 121-127`
- 前端证据链页：`frontend/src/components/EvidenceTimeline.vue:14-99`

第一步，后端把 `thinking_chain` 放到 `evidence_chain` 的 `strategy_selection` 或 `semantic_analysis` 里，前面已经讲过。

第二步，前端结果页在 `ResultPanel.vue:121-127` 用一个 `computed` 从 `props.result.evidence_chain` 里找出 `step === "strategy_selection"` 的那条记录，并读取 `entry.details.thinking_chain`。

第三步，`ResultPanel.vue:35-45` 把这段 CoT 做成单独的“决策智能体推理链路”折叠面板。也就是说，首页右侧结果区看到的大模型 CoT，并不是另起接口，而是从检测结果 JSON 中直接提取出来的。

第四步，证据链时间线组件 `EvidenceTimeline.vue` 会做第二层展示：

- `EvidenceTimeline.vue:73-75` 用 `hasThinkingChain(entry)` 判断当前证据项是否带有 `thinking_chain`
- `EvidenceTimeline.vue:33-49` 对包含 CoT 的证据项做特殊渲染
- `EvidenceTimeline.vue:35-42` 点击可展开查看完整 `thinking_chain`
- `EvidenceTimeline.vue:95-99` 会把 `thinking_chain` 从原始字段中剔除，再单独展示剩余 details，避免页面重复和混乱

所以你可以把整个 CoT 功能总结成一句话：

后端在 Agent 决策阶段把大模型推理链写入结构化证据链，前端再在结果页和时间线页分别做“重点展示”和“审计展示”，形成完整闭环。

### 6. 左侧检测历史功能是怎么做的

#### 6.1 数据库存储历史任务

#### 6.4 前端如何拉取和显示历史任务

前端数据拉取在两个地方：

这就实现了“左侧点历史任务，右侧切换展示详情”的联动。

#### 6.5 为什么历史任务能支持刷新和深链接

这两个小细节也值得讲。

## 十、建议你答辩时怎么讲

推荐你按“自顶向下”的方式讲，不要一上来就陷入某个函数细节。

### 第一层：先讲系统定位

可以先说：本项目不是单一分类模型，而是一个面向 Deepfake 人脸图像检测的完整辅助检测系统，重点是双智能体协作、结构化证据链、主动复核和多后端回退机制。

### 第二层：再讲系统流程

然后说：前端上传图片后，后端会创建异步任务，由特征提取智能体先做视觉证据生成，再由决策智能体做策略选择与最终融合判定，最后把结果、证据链和可视化产物返回给前端展示。

### 第三层：讲智能体设计

再强调：智能体的价值不在于“听起来高级”，而在于把系统拆成“取证”和“决策”两个清晰职责。这样比单模型直接输出标签更容易解释、更容易扩展，也更适合做异常处理。

### 第四层：讲可解释性与可靠性

最后说：本项目很重视可解释性和可靠性，因此不仅输出真假标签，还输出证据链、模型后端、阈值来源和复核原因；同时在模型缺失、语义超时、输入不适用等情况下，系统会自动回退并转人工复核，保证整体流程不中断。

## 十一、老师高概率追问问题与建议回答思路

### 1. 为什么你们要设计两个智能体，而不是一个模型直接分类？

建议回答：因为项目目标不仅是分类准确率，还要兼顾系统可解释性和工程闭环。特征提取智能体负责取证，决策智能体负责策略选择和结论生成，这样职责更清晰，便于扩展不同后端，也便于把证据链展示给用户和老师看。

### 2. 这个“智能体”到底体现在哪？

建议回答：体现在角色分工和决策流程上。系统不是把图片直接丢给一个黑箱，而是先产生结构化视觉证据，再基于任务画像选择策略，有条件地触发语义分析，最后输出结构化结论。它更接近“协作式处理流程”，而不是单点模型推理。

### 3. 上传图片后，后端为什么不直接同步返回结果？

建议回答：因为本系统是任务驱动的，包含多阶段处理和可视化产物生成。如果同步阻塞，一方面用户体验不好，另一方面不利于展示任务阶段和智能体执行过程。异步任务模式更符合系统化设计。

### 4. 决策智能体是不是可以随意决定最终结果？

建议回答：不是。它只能在预先定义好的三种策略中选择，最终标签也要经过阈值规则、质量条件、语义冲突条件等约束。这种设计是为了防止大模型自由发挥导致不可控结果。

### 5. 为什么要保留“需人工复核”，不直接二分类？

建议回答：因为 Deepfake 检测里存在灰区样本、低质量样本、多人场景和语义冲突样本。如果强行二分类，会把不确定性隐藏掉。保留人工复核更符合真实应用，也更体现系统可靠性设计。

### 6. 证据链和推理链有什么区别？

建议回答：证据链是系统正式返回的结构化审计记录，包括步骤、摘要和关键字段；推理链是其中某些 LLM 步骤额外返回的说明文本。项目对外主要强调结构化证据链，而不是完全依赖开放式思维链。

### 7. 前端结果页的热力图是怎么来的？

建议回答：热力图由特征提取智能体在视觉分析阶段生成，调用可视化服务把异常区域映射到人脸图像上，再作为任务产物保存并通过接口返回给前端展示。

### 8. 如果 LLM 不可用，系统是不是就不能工作？

建议回答：不会。LLM 不可用时，策略选择会回退到规则策略，语义分析也会回退到规则式语义后端，系统主流程仍然可以完成。这是项目设计时刻意保留的回退机制。

### 9. 模型权重文件和 metadata 文件为什么要分开？

建议回答：因为权重只负责推理本体，而 metadata 负责描述模型名称、数据版本、指标和阈值等部署信息。运行时需要同时知道“模型是谁”和“它该配什么阈值”，所以 metadata 也是运行时契约的一部分。

### 10. 当前仓库为什么还有 heuristic 后端？

建议回答：因为项目当前强调的是系统闭环和演示稳定性。真实模型资产未全部到位时，需要由 heuristic 后端兜底，保证上传、检测、结果展示和证据链输出仍然能稳定运行。

### 11. 结果页里显示的模型信息有什么意义？

建议回答：它能告诉使用者当前到底是哪个检测后端、视觉后端、融合后端和语义后端在工作，也能看到阈值来源。这保证了系统结果具备可追溯性，而不是一个来源不明的标签。

### 12. 数据库存任务结果有什么作用？

建议回答：一是支撑历史任务页和状态轮询；二是便于追溯；三是后续可以扩展成任务审计和样本复查。因为系统是面向“任务”而不是面向“一次性请求”设计的。

## 十二、适合结尾的总结口径

如果老师让你用一两句话概括整个系统，可以这样说：

本项目实现的不是单点模型调用，而是一个围绕 Deepfake 人脸图像检测构建的完整辅助检测系统。它通过特征提取智能体和决策智能体协同工作，把上传、检测、策略选择、语义辅助、融合判定、证据链输出和异常回退整合成一个可运行、可解释、可审计的工程闭环。
