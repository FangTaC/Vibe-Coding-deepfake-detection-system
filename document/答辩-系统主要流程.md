# 答辩-系统主要流程

## 一、先用一句话讲清楚这个系统在做什么

这个系统处理一张图片时，不是“上传后直接调用一个模型得出真假标签”这么简单，而是按一条完整任务链路来执行：

1. 前端上传图片，后端创建任务。
2. 后端异步启动检测流程，先做输入校验和任务状态更新。
3. 特征提取智能体读取图片、做人脸检测、逐脸分析视觉特征，并生成可视化产物。
4. 决策智能体根据提取到的结构化特征先选策略，再决定是否触发语义分析。
5. 系统把视觉特征、统计特征和可选语义分数送入融合模型，得到最终融合分数。
6. 根据阈值策略输出“疑似真实 / 疑似伪造 / 需人工复核”，并把结果、证据链、热力图、裁剪图、模型版本信息一并返回给前端。

如果老师问“你们检测一张图到底经历了什么”，你就可以沿着下面这条完整链路讲。

## 二、从上传一张图片开始，系统到底是怎么跑起来的

### 1. 前端上传图片后，后端先进入哪个接口

图片上传的后端入口在 [routes.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\api\routes.py) 第 32-41 行。

这一段做了三件事：

1. 接收前端传来的 `UploadFile`。
2. 读取文件二进制内容 `content = await file.read()`。
3. 调用 `detection_orchestrator.create_task(...)` 创建任务，再调用 `detection_orchestrator.submit_task(...)` 异步提交任务。

也就是说，用户点击上传后，并不是同步阻塞等待结果，而是先创建一个任务 ID，再由后端线程池异步执行检测流程。

如果老师问“为什么要先创建任务而不是直接返回结果”，可以回答：

> 因为系统不仅要完成图片判定，还要生成证据链、热力图、裁剪图、历史记录和状态轮询，所以更适合设计成任务制接口，而不是单次同步预测接口。

### 2. 上传后，图片先被保存到哪里

图片保存逻辑在 [storage.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\services\storage.py) 第 12-23 行。

- `create_task_directories()` 在第 12-17 行，会先为当前任务创建上传目录和产物目录。
- `save_upload()` 在第 19-23 行，会把这张图片真正写入磁盘。

也就是说，每个任务都会拥有自己独立的：

1. `uploads/<task_id>/原图文件`
2. `artifacts/<task_id>/可视化产物`

这也是后面热力图、标注图、人脸裁剪图可以按任务单独访问的基础。

### 3. 任务信息是怎么写进数据库的

任务创建后，`DetectionOrchestrator.create_task()` 会调用 `task_repository.create(...)`，代码在 [pipeline.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\services\pipeline.py) 第 33-38 行。

而 `TaskRepository.create()` 在 [repository.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\services\repository.py) 第 23-24 行，继续调用数据库层。

真正写入 SQLite 的逻辑在 [database.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\db\database.py) 第 56-81 行：

- 任务初始状态是 `queued`
- 初始阶段是 `queued`
- 初始进度是 `0.0`
- 同时保存 `filename`、`content_type`、`input_path`、时间戳等信息

数据库表的结构定义在 [database.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\db\database.py) 第 32-54 行。

所以，左侧“检测历史”之所以能显示，不是前端临时缓存，而是每一次检测任务都被持久化进 `tasks` 表。

## 三、任务创建后，异步检测流程怎么启动

### 1. 异步执行入口在哪里

任务提交入口在 [pipeline.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\services\pipeline.py) 第 77-81 行。

线程池初始化在 [pipeline.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\services\pipeline.py) 第 19-24 行。

这说明系统不是串行阻塞执行，而是把每张图片的检测任务投递到后台工作线程中执行。

### 2. 检测流程主函数在哪里

主检测流程在 [pipeline.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\services\pipeline.py) 第 40-75 行的 `run_task()`。

这个函数就是整张图片的后端主流程调度器，执行顺序非常清晰：

1. 从数据库取任务记录，拿到原图路径，第 41-44 行。
2. 实例化 `FeatureExtractionAgent` 和 `DecisionAgent`，第 45-46 行。
3. 更新任务进度到 `preprocessing` 和 `feature_extraction`，第 48-61 行。
4. 调用 `feature_agent.run(task_id, image_path)` 提取特征，第 62 行。
5. 更新任务进度到 `decision`，第 63-69 行。
6. 调用 `decision_agent.run(task_id, image_path, features)` 生成最终结果，第 70 行。
7. 把结果保存到数据库，第 71 行。
8. 如果中间异常，则记录失败信息，第 72-73 行。

所以你可以把 `run_task()` 理解为：**整个系统后端检测链路的总控函数。**

## 四、一张图片进入特征提取智能体后，具体做了什么

特征提取智能体定义在 [feature_extraction_agent.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\agents\feature_extraction_agent.py) 第 17-189 行。

### 1. 它先做什么

`FeatureExtractionAgent.run()` 的入口在 [feature_extraction_agent.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\agents\feature_extraction_agent.py) 第 25 行。

它的第一步是：

1. `load_image(image_path)` 读取图片，第 26 行。
2. `self.detector.detect(image)` 执行人脸检测，第 27 行。
3. 生成第一条证据链 `input_loaded`，记录输入图像尺寸、模式等信息，第 28-40 行。

### 2. 如果这张图一开始就没有检测到人脸，会怎么样

无脸分支在 [feature_extraction_agent.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\agents\feature_extraction_agent.py) 第 42-83 行。

这里系统会：

1. 在证据链中写入 `face_detection`，说明“未检测到有效人脸”，第 43-51 行。
2. 再写入 `feature_extraction`，说明“没有可用人脸特征”，第 52-60 行。
3. 调用 `save_annotated_image(task_id, image, [])` 生成一张无脸提示标注图，第 61 行。
4. 返回 `faces=[]`、`detector=none`、`visual=none` 的结构化结果，第 62-83 行。

这里很重要，因为这不是报错退出，而是进入一个“受限决策路径”。后面决策智能体会把它直接判成“需人工复核”。

### 3. 如果检测到了人脸，接下来做什么

正常分支在 [feature_extraction_agent.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\agents\feature_extraction_agent.py) 第 85-189 行。

核心流程是：

1. 收集所有人脸框 `boxes`，第 86 行。
2. 生成整张图的标注图 `annotated_original.jpg`，第 87 行。
3. 写入 `face_detection` 证据链，记录人脸数、检测后端、provider 和 bbox，第 90-103 行。
4. 遍历每一张人脸，第 105 行开始。
5. 对每张脸裁剪出人脸区域 `crop_face(...)`，第 107 行。
6. 生成对齐后人脸图 `face_x_aligned.jpg`，第 108 行。
7. 生成热力图 `face_x_gradcam.jpg`，第 109 行。
8. 调用视觉后端 `self.visual_backend.analyze(...)` 提取深伪分数及其他视觉指标，第 110 行。
9. 调用 `self.gallery.match(...)` 做公众人物图库匹配，第 111 行。
10. 把这一张脸的所有结果封装成 `FaceResult`，第 126-144 行。
11. 统一写入 `feature_extraction` 证据链，第 146-159 行。
12. 汇总原图标注图和每张脸的裁剪图、热力图，第 161-170 行。
13. 构造 `model_versions`，记录本次检测到底用了哪个检测后端、哪个视觉后端，第 172-176 行。

老师如果问“你们结果页里的热力图和裁剪图是哪里来的”，你可以直接答：

> 不是前端临时生成，而是在 `FeatureExtractionAgent.run()` 中逐脸生成的，其中裁剪图在第 108 行调用 `save_face_crop()`，热力图在第 109 行调用 `save_heatmap()`，原图标注图在第 87 行调用 `save_annotated_image()`。

### 4. 热力图、裁剪图和标注图的保存逻辑在哪

这些可视化产物的保存逻辑都在 [visualization.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\services\visualization.py)：

1. `save_annotated_image()` 在第 30-76 行
2. `save_face_crop()` 在第 79-82 行
3. `save_heatmap()` 在第 85-90 行

其中：

- 无人脸时，`save_annotated_image()` 会在底部写 “No face detected”，第 35-55 行。
- 有人脸时，会把每张脸框出来并标注 `Face 1`、`Face 2` 等，第 56-72 行。
- `save_heatmap()` 会先生成异常图，再叠加成热力图，第 85-89 行。

## 五、人脸检测到底是怎么做的，如果失败会怎么回退

### 1. 人脸检测总控入口在哪里

人脸检测的统一入口在 [backends.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\services\backends.py) 第 218-231 行的 `FaceDetectorManager`。

它的逻辑非常清楚：

1. 先尝试 `InsightFaceDetector`
2. 再尝试 `OpenCVHaarDetector`
3. 最后尝试 `HeuristicFaceDetector`

也就是说，人脸检测本身就是三级回退链。

### 2. 最优先的人脸检测方案是什么

第一优先级是 `InsightFaceDetector`，定义在 [backends.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\services\backends.py) 第 35-104 行。

它会：

1. 检查模型目录和依赖是否存在，第 42-56 行。
2. 优先尝试 CUDA provider，再尝试 CPU provider，第 58-79 行。
3. 成功后输出人脸框、关键点和 embedding，第 85-104 行。

这就是系统正常状态下最完整的人脸检测方案，因为它不仅能给框，还能给 embedding，方便后续做图库匹配。

### 3. 如果 InsightFace 不可用，会怎么回退

第二优先级是 `OpenCVHaarDetector`，定义在 [backends.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\services\backends.py) 第 107-167 行。

它会：

1. 动态导入 `cv2` 并加载 Haar cascade，第 113-125 行。
2. 对超大图先做缩放，第 130-137 行。
3. 用两组逐步放宽的参数尝试检测，第 140-148 行。
4. 把检测框按比例映射回原图，第 151-156 行。
5. 自动生成默认关键点和 histogram embedding，第 157-165 行。

所以，即使深度学习人脸检测器不可用，系统仍然能用传统方法继续工作。

### 4. 如果 OpenCV 也不行，还会怎样

最后会退到 `HeuristicFaceDetector`，定义在 [backends.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\services\backends.py) 第 170-215 行。

它的思路是：

1. 先构造中心框、四角框、左右半幅等候选框，第 175-184 行。
2. 对每个候选框计算简单的 face-like probability，第 185-188 行。
3. 如果整体最高分都不到 0.60，则认为无人脸，第 190-191 行。
4. 再按概率、阈值和 IOU 去重筛选有效候选框，第 193-201 行。
5. 最后把这些候选框封装为伪“检测结果”，第 203-214 行。

所以你可以概括为：

> 人脸检测模块不是单点依赖，而是 InsightFace 失败后自动回退 OpenCV，再失败回退启发式检测，直到彻底无法找到可用人脸时才进入人工复核分支。

## 六、视觉模型是怎么调用的，如果训练好的轻量化模型不可用会怎样

### 1. 视觉后端统一入口在哪里

视觉分析统一入口在 [backends.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\services\backends.py) 第 320-338 行的 `VisualBackendManager`。

逻辑是：

1. 优先调用 `EfficientNetVisualBackend`
2. 如果失败，就回退到 `HeuristicVisualBackend`

### 2. 训练好的轻量化视觉模型是怎么被调用的

训练好的轻量化模型就是 `EfficientNet-B0`，加载和推理逻辑在 [backends.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\services\backends.py) 第 234-296 行。

具体流程是：

1. `_load_model()` 先检查权重文件和 metadata 是否存在，第 242-248 行。
2. 动态创建 `efficientnet_b0(weights=None)` 网络结构，第 253 行。
3. 替换最后分类层为二分类头，第 254-255 行。
4. 加载训练好的 `state_dict`，第 256-258 行。
5. 在 `analyze()` 中对人脸图做 `Resize(256,256)`、`ToTensor()`、`Normalize()`，第 271-277 行。
6. 模型输出 softmax 概率，第 279-281 行。
7. 同时计算启发式视觉统计量 `_estimate_fake_probability(...)`，第 282 行。
8. 最终不是直接用模型概率，而是做一个校准融合：`0.7 * CNN概率 + 0.3 * 启发式概率`，第 284 行。

这一点很值得讲，因为它说明系统不是把训练模型生硬塞进去，而是做了工程化校准。

### 3. 如果 EfficientNet-B0 不可用，会怎样

如果视觉模型权重缺失、metadata 缺失、`torch` 导入失败，`EfficientNetVisualBackend.analyze()` 就会抛异常；然后 `VisualBackendManager.analyze()` 在第 325-329 行会自动退回 `HeuristicVisualBackend`。

回退后端定义在 [backends.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\services\backends.py) 第 299-317 行。

它会：

1. 调用 `_estimate_fake_probability(...)` 计算启发式伪造分数，第 302-303 行。
2. 结合增强一致性分数再做一次加权，第 304-305 行。
3. 输出 `deepfake_score`、`quality_score`、`dct_score`、`compression_score` 等结构化视觉特征，第 306-317 行。

也就是说，**即使训练好的轻量化视觉模型不可用，系统仍然会继续生成可解释的视觉分数，而不会中断。**

## 七、特征提取完后，为什么还要进入决策智能体

决策智能体定义在 [decision_agent.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\agents\decision_agent.py) 第 27-382 行。

### 1. 它的第一步是什么

`DecisionAgent.run()` 在第 35-173 行。

进入后，它先做三件事：

1. 取出 `faces`、`evidence`、`artifacts`、`model_versions` 等中间结果，第 41-45 行。
2. 从 `model_registry.resolve_threshold_policy()` 读取当前阈值策略，第 46 行。
3. 如果 `faces` 为空，直接走无脸决策 `_build_no_face_result()`，第 48-56 行。

### 2. 阈值是在哪里设定的

阈值解析逻辑在 [model_registry.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\services\model_registry.py) 第 42-69 行。

优先级是：

1. 优先读取融合模型 metadata 里的 `thresholds`
2. 如果融合模型没有，再读视觉模型 metadata 里的 `thresholds`
3. 都没有时，退回 `settings.true_threshold` 和 `settings.fake_threshold`

也就是说，系统运行时用的阈值不是写死在某个页面上，而是由模型资产中的 metadata 优先控制。

### 3. 什么叫任务画像，为什么要先做这个

任务画像构建在 [decision_agent.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\agents\decision_agent.py) 第 256-282 行。

这里会聚合出：

1. `face_count`
2. `max_face_score`
3. `mean_face_score`
4. `quality_score`
5. `public_figure_candidates`
6. `conflict_level`
7. `need_semantic_check`

其中 `need_semantic_check` 的判定很关键，第 268-273 行规定，只要满足以下任意条件，就认为需要进一步语义分析：

1. 最高人脸分数落在真图阈值和伪造阈值之间
2. 图片里有多张脸
3. 检测到了公众人物候选
4. 视觉冲突程度 `conflict_level >= 0.35`

所以系统不是每次都调用语义模型，而是先根据任务画像做策略选择。

## 八、策略选择是怎么做的，如果大模型不可用怎么办

### 1. 策略选择入口在哪里

策略选择在 [decision_agent.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\agents\decision_agent.py) 第 284-300 行的 `_select_strategy()`。

它的逻辑是：

1. 如果 `QwenStrategyBackend` 可用，优先让 LLM 选策略，第 288-292 行。
2. 如果 LLM 失败，则退回规则策略，第 293-296 行。
3. 如果一开始 LLM 就不可用，也直接走规则策略，第 297-300 行。

### 2. 大模型策略后端在哪里

`QwenStrategyBackend` 定义在 [backends.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\services\backends.py) 第 634-697 行。

它会把：

1. 任务画像
2. 当前真图阈值
3. 当前伪造阈值

一起发给大模型，请它返回结构化 JSON，包括：

1. `selected_strategy`
2. `reason_codes`
3. `semantic_required`
4. `expected_risk_level`
5. `thinking_chain`

也就是说，前端结果页里看到的 CoT/思维链，不是手写说明，而是策略选择大模型返回的结构化内容。

### 3. 如果策略大模型不可用，会怎么回退

规则回退逻辑在 [decision_agent.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\agents\decision_agent.py) 第 302-331 行。

规则很清楚：

1. 如果图像质量低于 `low_quality_threshold`，直接选 `review_only`，第 303-309 行。
2. 如果任务画像判断需要语义检查，选 `visual_plus_semantic`，第 310-325 行。
3. 否则，直接选 `visual_only`，第 326-330 行。

这说明即使策略 LLM 全部不可用，系统仍能继续完成检测决策。

## 九、语义分析是怎么接入的，如果失败会怎样

### 1. 什么时候会触发语义分析

只有当 `strategy_decision.semantic_required == True` 时，系统才会进入语义分析，代码在 [decision_agent.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\agents\decision_agent.py) 第 88-112 行。

进入后，系统会：

1. 调用 `self.semantic_backend.analyze(image_path, semantic_context)`，第 90 行。
2. 读取 `semantic_flags`、`semantic_score`、`summary`，第 91-99 行。
3. 把语义证据链加入结果，第 101-112 行。

### 2. 语义分析后端有哪些层级

`SemanticBackendManager` 在 [backends.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\services\backends.py) 第 595-631 行。

优先级是：

1. `QwenVisionSemanticBackend`，即远程视觉大模型 API，第 603-610 行
2. `LocalQwenSemanticBackend`，即本地视觉语言模型，第 612-616 行
3. `RuleBasedSemanticBackend`，即规则语义回退，第 617-631 行

### 3. 如果语义模型超时了，会怎样

如果本地语义模型超时，`SemanticBackendManager` 会：

1. 调用规则语义后端生成回退结果，第 617-618 行
2. 额外加一个 `SEMANTIC_TIMEOUT` 标记，第 619-627 行
3. 把 summary 改成“已自动回退到规则语义分析”，第 628 行

## 十、融合模型是怎么参与最终决策的

### 1. 融合特征是怎么拼出来的

融合特征拼接在 [decision_agent.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\agents\decision_agent.py) 第 333-344 行。

送入融合模型的 8 维特征分别是：

1. `max_face_score`
2. `mean_face_score`
3. `mean(dct_score)`
4. `mean(quality_score)`
5. `mean(augmentation_consistency_score)`
6. `mean(compression_score)`
7. `face_count`
8. `semantic_score`

### 2. 融合模型怎么调用训练好的 XGBoost

融合模型优先调用 `XGBoostFusionBackend`，定义在 [backends.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\services\backends.py) 第 341-374 行。

逻辑是：

1. `_load()` 检查 XGBoost 模型文件和 metadata 是否存在，第 349-365 行。
2. 存在则加载 `Booster`，第 359-362 行。
3. `predict()` 中把特征包装成 `DMatrix`，再预测一个 0 到 1 的融合分数，第 367-373 行。

### 3. 如果 XGBoost 不可用，会怎样

如果模型缺失、依赖缺失或加载失败，`FusionBackendManager.predict()` 在 [backends.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\services\backends.py) 第 399-403 行会自动回退到 `HeuristicFusionBackend`。

启发式融合定义在 [backends.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\services\backends.py) 第 376-391 行。

它会把 8 维特征按经验权重线性组合，例如：

1. `max_face_score * 0.38`
2. `mean_face_score * 0.18`
3. `dct_mean * 0.10`
4. `(1 - quality_mean) * 0.12`
5. `(1 - consistency_mean) * 0.08`
6. `compression_mean * 0.06`
7. `face_count * 0.03`
8. `semantic_score * 0.05`

再裁剪到 0 到 1 区间。

所以老师如果问“融合模型没了系统还能不能跑”，你可以答：

> 可以。融合层本身也有训练模型和启发式方案两级机制，XGBoost 不可用时会自动退回启发式融合，不会导致系统中断。

## 十一、最终标签是怎么判出来的，阈值没达到会发生什么

最终标签判定逻辑在 [decision_agent.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\agents\decision_agent.py) 第 346-369 行的 `_finalize_label()`。

### 1. 最终可能输出哪些标签

系统标签定义在 [task.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\schemas\task.py) 第 26-29 行：

1. `疑似真实`
2. `疑似伪造`
3. `需人工复核`

### 2. 哪些情况会直接输出“需人工复核”

在 `_finalize_label()` 里，以下情况会直接输出 `review_required`：

1. 策略本身就是 `review_only`，第 357-358 行
2. 图像质量过低，第 359-360 行
3. 语义模型超时，第 361-362 行
4. 视觉和语义证据冲突，且当前分数又不够绝对确定，第 363-364 行
5. 融合分数落入灰区，第 369 行

### 3. 阈值到底怎么参与最终判定

判定逻辑非常直接：

1. 如果 `fusion_score <= true_threshold`，输出 `疑似真实`，第 365-366 行
2. 如果 `fusion_score >= fake_threshold`，输出 `疑似伪造`，第 367-368 行
3. 如果在二者之间，输出 `需人工复核`，第 369 行

这就是你答辩时要反复强调的“三态输出”机制。

也就是说，**不是所有图片都被硬判成真或假**。系统承认灰区样本的存在，并把不确定样本安全地送去人工复核。

### 4. 如果没达到阈值，系统会输出什么

“没达到阈值”这里要分两种理解：

1. 如果没有达到伪造阈值、但又高于真图阈值，也就是落在灰区，系统会输出 `需人工复核`，并附带 `review_reason = 检测分数落入灰区，需要人工复核。`，对应 [decision_agent.py](F:\AgentDeepfakeFaceDet\AgentDeepfakeFaceDet\backend\app\agents\decision_agent.py) 第 369 行。
2. 如果分数足够低，低于真图阈值，那并不是“没达到阈值”，而是已经满足“疑似真实”的条件，会输出 `likely_real`，对应第 365-366 行。

所以准确说法是：

> 系统使用双阈值而不是单阈值。低于真图阈值判为疑似真实，高于伪造阈值判为疑似伪造，中间灰区统一进入人工复核。

## 十三、如果老师要你把“完整检测一张图”的流程一句一句讲出来，可以这样讲

可以直接按下面这段顺着说：

> 用户上传一张图片后，系统先进入 `routes.py` 的 `POST /tasks` 接口，读取图片内容并交给 `DetectionOrchestrator` 创建任务；随后图片被保存到任务专属目录，同时任务信息写入 SQLite 数据库。异步线程池开始执行 `run_task()`，先更新任务进度，再调用 `FeatureExtractionAgent` 读取原图、做人脸检测、逐脸裁剪、调用视觉后端提取深伪分数与频域、质量、一致性等特征，同时生成标注图、裁剪图和热力图。如果 InsightFace 不可用，会回退到 OpenCV Haar；如果再不可用，会回退到启发式人脸检测。视觉分析同样优先使用训练好的 EfficientNet-B0，若权重或依赖不可用，则回退到启发式视觉评分。之后进入 `DecisionAgent`，先读取阈值策略，构建任务画像，再由大模型或规则系统选择策略，决定是否需要语义分析。如果需要，就调用远程或本地语义模型；如果语义模型不可用或超时，则回退到规则语义分析。最后系统把视觉分数、统计特征和语义分数送入 XGBoost 融合模型；若融合模型不可用，则回退到启发式融合。得到融合分数后，系统根据双阈值输出“疑似真实”“疑似伪造”或“需人工复核”。最终检测结果、证据链、模型版本信息、热力图和人脸裁剪图一起写入数据库，并通过结果接口返回给前端展示。

## 十四、最后帮你压缩成老师最常问的几个关键点

### 1. 这个系统是不是只调用了一个模型

不是。它至少包含：

1. 人脸检测模块
2. 视觉特征提取模块
3. 策略选择模块
4. 可选语义分析模块
5. 融合判别模块
6. 证据链与可视化输出模块

### 2. 如果模型缺失，系统会不会直接报错

不会。人脸检测、视觉分析、语义分析、融合判别都设计了回退机制。

### 3. 最终标签是不是简单看 0.5

不是。系统使用双阈值，低于真图阈值判真实，高于伪造阈值判伪造，中间灰区统一人工复核。

### 4. 为什么你们能展示热力图和检测历史

因为产物会按任务保存到磁盘，任务结果会按 JSON 持久化到 SQLite，前端再通过 `/tasks`、`/result`、`/artifacts` 这些接口读取展示。

### 5. 这个系统最体现工程能力的地方是什么

不是单一模型效果，而是：

1. 任务制异步执行
2. 多级回退机制
3. 三态输出和双阈值策略
4. 证据链与思维链联动
5. 可视化产物自动生成
6. 数据库存储和历史追踪
